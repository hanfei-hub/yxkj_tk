# 后续开发计划

当前项目已经从 MVP 进入正式可用版本建设阶段。后续开发应围绕“数据采集 -> AI 选品 -> 老师审核 -> 权重学习 -> 1688 货源匹配 -> 前端展示”的完整闭环推进。

## 阶段 1：稳定基础链路

目标：

- 保证 FastMoss 同步、标题翻译、商品展示稳定。
- 保证前端不会因 token、图片、网络异常卡死。

任务：

- 完善 FastMoss 同步日志展示。
- 前端页面刷新按钮稳定读取数据库最新数据。
- 图片加载继续异步化和缓存优化。
- token 失效统一提示“请重新登录”。
- 确认所有页面在未登录状态下仍能加载基础 UI。

验收：

- 同步 10 个 FastMoss 商品后，前端能展示图片、中文标题、日元价格和销量。
- 本地网络异常时，前端不崩溃。

## 阶段 2：衍生品生成稳定化

目标：

- 每个 FastMoss 原商品稳定生成 10 个衍生品。
- 每个衍生品都有 8 个维度报告。

任务：

- 清理衍生品提示词中的历史编码问题。
- 明确模型输出格式。
- 生成失败时记录失败原因。
- 支持按原商品逐条异步生成。
- 前端展示衍生品生成状态。

验收：

- 同步 10 个原商品后，理论上生成 100 个衍生品。
- 每条衍生品能查到 `source_product_id`、`family_id` 和 8 个维度报告。

## 阶段 3：老师审核和权重学习完善

目标：

- 老师拒绝数据能稳定反向影响商品分族权重。

任务：

- 完善衍生品审核窗口样式。
- 拒绝原因下拉框固定展示 8 个维度。
- 审核后刷新当前卡片状态。
- 增加权重变化可视化页面。
- 增加商品分族详情页，展示该分族 8 个维度权重。

验收：

- 老师拒绝某衍生品的“商品周期性”后，该分族 `dimension_2` 权重降低。
- 权重总和保持 100%。
- 同分族再次生成衍生品时，提示词包含新的权重。

## 阶段 4：1688 API 正式接入

目标：

- 衍生品自动匹配具体可采购 1688 商品。

任务：

- 根据第三方正式 API 文档完成参数适配。
- 配置搜索接口路径、请求方法、字段映射。
- 支持分页拉取候选商品，最多 200 个。
- 调用大模型基于“衍生品名称 + 1688 首图 URL”评分。
- 阈值默认 90%，后台可配置。
- 匹配成功后写回 1688 图片、价格、销量、店铺和链接。
- 匹配失败时记录最高分和失败原因。

验收：

- 对待匹配衍生品执行批量匹配后，匹配成功的商品状态为 `matched`。
- 前端衍生品卡片能展示 1688 商品图片和价格。

## 阶段 5：前端体验打磨

目标：

- 桌面程序达到接近网页级视觉和交互标准。

任务：

- 统一商品卡尺寸、图片区域、价格样式、销量样式。
- 教师看板和智能选品商品卡保持一致。
- 衍生品审核窗口优化滚动、卡片排列、图片展示。
- 主题皮肤继续完善，默认浅色。
- 左下角登录用户信息美化。
- 加载态、空状态、错误态统一。

验收：

- 常用分辨率下无错位、无重叠、无明显卡顿。
- 图片比例稳定，不出现宽度不一致。

## 阶段 6：正式发布和运维

目标：

- 形成可交付、可部署、可恢复的正式版本。

任务：

- 固定打包产物名称。
- 增加版本号。
- 增加启动日志。
- 增加服务器定时任务监控。
- 增加数据库备份策略。
- 建立异常告警方案。
- 整理交付文档和操作视频脚本。

验收：

- 新机器可按文档安装运行。
- 服务器重启后后端和定时任务自动恢复。
- 数据库可备份和恢复。
